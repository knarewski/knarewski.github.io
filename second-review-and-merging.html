Second review and merging

The reviewer cross-checks new code against his comments, marking the addressed ones as “resolved”, and optionally skims the entire code again. If everything is in order, the Pull Request can be approved.

The developer uses git rebase -i --autosquash master, to merge fixups into their corresponding commits, then pushes the changes. If there were any difficult conflicts, it’s sometimes worth asking for another quick review to make sure nothing got accidentally broken in the meantime. After that’s completed, the code should be merged using rebase strategy.

Rules

  • After getting approval autosquash the fixups which address the suggestions.
  • Don’t hesitate to dismiss review and ask for one last look if there were some significant conflicts when squashing.
  • (😺) When everything is good to go, merge using Rebase and merge button

Caveats

  • This approach assumes certain degree of trust and competence. It would be possible for a developer to smuggle some undesired code on that stage. If that’s a problem, a review after squashing can be enforced.

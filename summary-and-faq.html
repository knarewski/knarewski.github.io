FAQ

Q: Why not just squash all the commits into one when merging?

A: Git history is a powerful tool which helps understanding past changes, including the authorâ€™s reasoning and scope of change. Mixing preparatory changes like style fixing, adding missing test coverage and performing initial refactor makes it much harder to use.

Q: What if I need to revert a feature?

A: Thanks to rebase strategy, all the new commits are next to each other in git history, which allows to revert range of commits using git revert OLDER_COMMIT^..NEWER_COMMIT

Q: Is it really worth the effort?

A: For me it is, the flow may feel a bit complex at the beginning, but it gets natural with time. The most difficult bit is squashing conflicts, sometimes it is better to just be pragmatic and add a new commit on top.
